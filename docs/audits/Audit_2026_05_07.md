### 2.4 Code quality

**Consistent and disciplined across all crates.** `unsafe_code = "forbid"` and
`missing_docs = "warn"` are honored at the workspace level. All public items
carry doc comments. The structured error taxonomy is complete and consistent.
No `unwrap()` / `expect()` outside tests.

**Specific findings:**

1. **`crates/pureflow-cli/src/main.rs` is a 1309-line monolith with hand-rolled
   argument parsing** (lines 142â€“192, `parse_run_command` at 290). The
   `command_output` function dispatches on raw `&[String]` slices via `match`.
   This works for four subcommands, but:
   - Error messages for malformed arguments (`CliError::Usage`) print the
     entire usage string with no indication of which argument was wrong.
   - Shell completion is not possible without `clap` or a similar framework.
   - Five new epics (`cdt-dtx`, `cdt-2av`, `cdt-4be`, `cdt-a1s`, `cdt-jrf`)
     will each add subcommands or flags; the monolith will grow proportionally.
   Bead `cdt-dtx.1` (cli-migrate-to-clap) directly addresses this.

2. **`crates/pureflow-core/tests/public_api_substrate.rs:6â€“15` â€” leak detection
   by source-text pattern matching is fragile.** The patterns ` Runtime`,
   ` SendPermit`, ` RecvError`, ` SendError` will match any type or function with
   those names, not only `asupersync` ones. A false positive would be `pub
   struct WorkflowRunSummary` if it had a field called `runtime: SomeRuntime`.
   A false negative would be any re-export that uses a type alias. The test
   passes today and the intent is right, but a symbol-level check (e.g.,
   `cargo-public-api` or type-path-aware inspection) would be more robust if
   the leak surface ever widens.

3. **`pureflow-introspection` has 4 tests for 938 lines of code** â€” the thinnest
   unit-test coverage in the workspace relative to crate size. The `inspect`
   and `explain` CLI commands both route through this crate. The snapshot tests
   in `pureflow-wasm` and `pureflow-cli` provide some indirect coverage via CLI
   integration, but the projection logic in `introspect_workflow` and
   `workflow_introspection_to_json_string` is exercised only at the surface.
   A regression in field ordering, enforcement-level mapping, or schema-ref
   serialization would not be caught by the four existing tests alone.

4. **`pureflow-wasm` tests take ~10 seconds** because they compile the uppercase
   guest fixture at test time. This is the correct design (real integration
   coverage), but it means the test suite has a slow-path that could grow as
   more WASM fixtures are added. No action required now; noting so the first
   additional WASM fixture evaluates whether a separate test profile or cached
   artifact is warranted.

5. **`pureflow-workflow-format` has YAML and TOML parsing in workspace
   dependencies (`serde_yml`, `toml`) but the CLI only accepts JSON workflow
   files.** The parsing crate exists; it's only the CLI dispatch that is
   missing. Bead `cdt-dtx.2` (cli-load-toml-and-yaml-workflows) is planned.
   No finding â€” noting the gap is smaller than it appears.

6. **`asupersync` is pinned to `0.2.9` with a caret requirement** â€” the
   workspace `Cargo.toml` declares `asupersync = "0.2.9"`. This allows any
   `0.2.x` compatible update. Since `asupersync` is experimental, a `=0.2.9`
   exact pin would be more conservative. The `Cargo.lock` pins the resolved
   version; the key question is whether minor updates should be absorbed
   automatically. No action required, but worth a conscious decision before
   locking down for a release candidate.

**No identified security vulnerabilities.** WASM guest code is sandboxed via
Wasmtime fuel limits, and host-owned output validation runs before packets
enter downstream edges. Workflow JSON parsing uses `serde_json` against typed
structures. No attacker-controlled inputs reach format strings or process
invocations.

**No identified algorithm-level inefficiencies.** The metadata capture path,
backpressure channels, and workflow orchestration all use appropriate
primitives. Criterion benchmarks will eventually surface real baselines.

## 3. Recommendations

Ordered by leverage.

1. **Create a CI workflow** (bead `cdt-2av.1`). The validation matrix is
   complete and documented; automating it in GitHub Actions (or equivalent) is
   a one-time 30-minute task that prevents silent regressions indefinitely.
   This is the single highest-leverage action remaining for the project.

2. **Migrate CLI to `clap`** (bead `cdt-dtx.1`). The five open epics will each
   add commands or flags. Doing the migration now, while the surface is four
   subcommands and two flags, is far cheaper than after the CLI doubles in size.

3. **Commit working-tree changes before opening the next bead.** The closed
   `cdt-pl3.5` and `cdt-pmb.*` work is sitting uncommitted in nine files. Run
   `jj commit` (or equivalent) to close the snapshot cleanly, then start the
   next bead fresh.

4. **Expand `pureflow-introspection` test coverage.** The crate is 938 lines
   and powers two CLI commands. Add at minimum: a test for
   `introspect_workflow` error paths (unknown node reference in a contract,
   mismatched port direction), and a snapshot test for the JSON serialization
   of a workflow with mixed native and WASM nodes. This is a natural companion
   to bead `cdt-a1s.1` (metadata snapshot tests) â€” consider doing both in
   the same bead or sequentially.

5. **Capture Criterion baselines** (bead `cdt-a1s.5`). Both benchmarks compile
   and build cleanly. Running them once and committing the `target/criterion/`
   output (or a summary in `docs/`) makes the first perf regression visible.

6. **Replace the public API substrate text scan with a path-aware check.** The
   current `starts_public_item` heuristic would false-positive on any type
   whose name contains `Runtime`, `SendPermit`, etc. As the codebase grows,
   so does the risk. Options: add an allowlist of acceptable matches, or adopt
   `cargo-public-api` if the project adds more complex type surfaces.

7. **Decide the `asupersync` version pinning strategy before release.** Either:
   - Keep `asupersync = "0.2.9"` (semver-compatible patch updates absorbed
     automatically) â€” acceptable if the crate's SEMVER discipline is trusted.
   - Change to `asupersync = "=0.2.9"` (exact pin) â€” appropriate for a
     research-vehicle runtime where stability is uncertain.
   The `Cargo.lock` protects any single checkout; this only matters for
   reproducibility across fresh checkouts or after `cargo update`.

8. **Add supervisor failure matrix tests** (bead `cdt-a1s.2`). The engine has
   `WorkflowWatchdogPolicy`, deadlock detection, and node-error propagation.
   Tests that verify the watchdog fires on a genuinely stalled graph and that
   a failing node does not orphan downstream tasks would make these mechanisms
   observable and prevent regression.

9. **Exercise the YAML/TOML workflow paths in `pureflow-workflow-format`.**
   The YAML and TOML dependencies exist and `serde_yml`/`toml` are wired into
   the workspace, but no tests cover those parse paths. At minimum, one round-
   trip test per format before bead `cdt-dtx.2` closes.
