## Recent Work

The May 6 JJ history shows the WASM productization stream completed through
`cdt-pl3.5`:

- `cdt-pl3.2`: wasm-component-abi-conformance
- `cdt-pl3.3`: wasm-output-contract-validation
- `cdt-pl3.4`: wasm-cancellation-and-limits
- `cdt-pl3.5`: real-wasm-native-mixed-pipeline

`pureflow-wasm` now has real Wasmtime/component ABI coverage against the
`wasm32-wasip2` uppercase fixture, including QuickCheck-style property coverage
for generated byte batches and metadata preservation.

`pureflow-engine` now exposes `BatchNodeExecutor<E>`, which adapts topology-blind
`BatchExecutor` implementations into graph nodes by draining `PortsIn`, invoking
the batch executor, and sending all returned packets through normal `PortsOut`
validation. This is the host-owned validation boundary used for WASM outputs.

`pureflow-wasm` now has `WasmtimeExecutionLimits` and cancellation-aware
invocation:

- per-invocation fuel is configured on the Wasmtime store
- synchronous guest invocation can be interrupted by a cancellation watcher
- pre-cancelled invocations return stable cancellation errors
- fuel exhaustion maps to stable execution errors

`crates/pureflow-wasm/examples/mixed_pipeline.rs` now runs a real bounded graph:

- native source sends bytes with normal `PortsOut`
- middle node uses `WasmtimeBatchComponent` wrapped by `BatchNodeExecutor`
- native sink receives transformed bytes with normal `PortsIn`
- the example builds the uppercase fixture into a temporary `wasm32-wasip2`
  artifact before running

The example command is documented in
`crates/pureflow-wasm/examples/README.md`:

```sh
nix develop . --command cargo run -p pureflow-wasm --example mixed_pipeline
```
