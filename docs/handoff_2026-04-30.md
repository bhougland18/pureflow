## Recent Work

`pureflow-core` now has a tiered metadata policy adapter:

- `MetadataTier` names control, data, and high-cost data metadata classes.
- `TieredMetadataPolicy::control_only()` records control facts and drops data-tier records by default.
- `TieredMetadataPolicy::sample_data_every(...)` samples data-tier facts with a deterministic counter.
- `TieredMetadataPolicy::with_high_cost_data()` explicitly opts in to high-cost data records.
- `TieredMetadataSink<S>` wraps any existing `MetadataSink` without changing the `MetadataSink` trait.
- `TieredMetadataSink::record(...)` treats current `MetadataRecord` values as control facts because they do not carry payload bodies.
- `TieredMetadataSink::record_with_tier(...)` lets future payload-shape or Arrow metadata call sites apply data/high-cost policy explicitly.
- Tests cover default control forwarding, default data/high-cost dropping, deterministic sampling, and explicit high-cost opt-in.

`pureflow-cli` now has first-pass `run` and `explain` commands:

- `pureflow explain <workflow.json>` validates the workflow and prints topology, scaffold execution mode, and metadata policy details.
- `pureflow run <workflow.json> <metadata.jsonl>` validates the workflow, runs it through the scaffold no-op executor, and writes lifecycle metadata JSONL.
- CLI `run` uses `AsupersyncRuntime`, `pureflow-engine::run_workflow_with_metadata_sink`, `JsonlMetadataSink`, and `TieredMetadataSink`.
- `run_workflow_with_metadata_sink` was added to `pureflow-engine` so workflow-level execution can emit runtime metadata without changing `NodeExecutor`.
- The CLI still does not load real node implementations; `run` is currently a scaffold execution path that proves validated topology, runtime scheduling, and metadata capture.
- Tests cover `explain` output, JSONL metadata capture, and engine lifecycle metadata recording.

`pureflow-runtime` now has an optional tracing bridge:

- workspace dependency added as `tracing = "0.1.44"`
- crate feature added as `pureflow-runtime/tracing`
- lifecycle observations emit `tracing::info!` events when the feature is enabled
- lifecycle metadata records emit `tracing::debug!` events at the metadata seam
- pre-run cancellation observations emit `tracing::warn!` events
- tracing labels are stable snake-case strings and do not replace typed metadata records
- default builds compile no-op trace shims and keep tracing emission disabled

`pureflow-core` now has a Criterion metadata overhead benchmark:

- workspace dependency added as `criterion = "0.8.2"`
- `crates/pureflow-core/benches/metadata_overhead.rs` benchmarks no-op, JSONL/default, and tiered metadata sink paths
- the benchmark includes control-record forwarding, JSONL serialization to `io::Sink`, tiered no-op forwarding, tiered JSONL forwarding, default data-tier dropping, and sampled data-tier forwarding
- the bench target is configured with `harness = false` and compiles through `cargo bench -p pureflow-core --bench metadata_overhead --no-run`

`pureflow-engine` now has an executor registry boundary:

- `NodeExecutorRegistry` resolves workflow node IDs to concrete executors.
- `SingleNodeExecutorRegistry` preserves the existing one-executor-for-all-nodes API.
- `StaticNodeExecutorRegistry` provides an in-memory node-to-executor map for tests and simple callers.
- `run_workflow_with_registry(...)` and `run_workflow_with_registry_and_metadata_sink(...)` execute workflows through the registry path.
- Existing `run_workflow(...)` and `run_workflow_with_metadata_sink(...)` are compatibility wrappers over the single-executor registry.
- Tests cover per-node executor resolution and missing registry entries.

`pureflow-engine` now has a workflow run summary API:

- `WorkflowTerminalState` reports completed, failed, and cancelled terminal states.
- `WorkflowRunSummary` tracks scheduled, completed, failed, cancelled, observed-message, and error counts plus the first observed error.
- `run_workflow_with_registry_summary(...)` and `run_workflow_with_registry_and_metadata_sink_summary(...)` return summaries for registry-based execution.
- `run_workflow_summary(...)` and `run_workflow_with_metadata_sink_summary(...)` preserve the existing single-executor convenience path while returning summaries.
- Existing `Result<()>` workflow execution APIs are compatibility wrappers that convert summaries back into first-error results.
- Node execution failures and cancellations are now represented as summary data by the summary APIs; executor resolution errors still return `Err(...)`.
- `observed_message_count` intentionally remains zero until the queue-pressure/message accounting observability beads attach message observations to the runner.
- Tests cover successful summaries, first-error retention without returning `Err(...)`, and cancellation terminal summaries.

`pureflow-engine` now has an observer-aware workflow execution surface:

- `run_workflow_with_registry_and_observers_summary(...)` is the core registry workflow path, accepting a `LifecycleHook` plus `MetadataSink` and returning `WorkflowRunSummary`.
- `run_workflow_with_registry_and_observers(...)` provides the legacy `Result<()>` shape over the same observer-aware path.
- `run_workflow_with_observers_summary(...)` and `run_workflow_with_observers(...)` preserve the single-executor convenience API while accepting observer sinks.
- The no-observer and metadata-only workflow APIs are now wrappers over the observer-aware path using `NoopLifecycleHook` and/or `NoopMetadataSink`.
- Existing lifecycle and port metadata behavior stays in `pureflow-runtime::run_node_with_observers`; the engine now wires workflow-level scheduling, registry resolution, cancellation, ports, summaries, and observers through one implementation.
- Tests cover observer-aware summary execution recording both lifecycle hook events and lifecycle metadata records.

`pureflow-engine` now has contract-aware output validation:

- `pureflow-core::PortsOut` can attach an `OutputPacketValidator` that runs before a reserved send permit commits a packet into downstream graph edges.
- `PortSendError::Rejected` represents pre-send validation failures and prevents rejected packets from being enqueued.
- `WorkflowOutputContracts` derives the workflow-runner output subset from `NodeContract` metadata without moving contract logic into port transport.
- Contract-aware workflow APIs were added for registry and single-executor paths, including observer-aware summary variants.
- The engine validator checks that output packets carry the current workflow ID, execution metadata, and a source endpoint matching the node/output port contract.
- Tests cover core pre-enqueue rejection, accepted contract-aware output packets, and rejected mismatched packet sources.

`pureflow-cli run` now exercises a real native registry path:

- The old scaffold no-op executor was replaced with a per-node `StaticNodeExecutorRegistry<CliNativeExecutor>`.
- `pureflow run` now executes through `run_workflow_with_registry_and_metadata_sink(...)`, so the CLI path uses the same registry, graph wiring, bounded ports, lifecycle metadata, and message metadata as engine callers.
- Each CLI native executor drains its declared inputs until upstream closure, counts received packets, and emits one deterministic packet per declared output port.
- Output packet metadata includes workflow ID, execution metadata, source endpoint, and a route target derived from the workflow edge map.
- `pureflow explain` now reports `execution: native-registry` and lifecycle, message, plus queue-pressure metadata records.
- CLI tests assert the native-registry explanation, reproducible JSONL metadata, lifecycle records, message records, queue-pressure records, and the expected 13-record run output for the linear fixture.

The repo now has a runnable native linear ETL example:

- `examples/native-linear-etl.workflow.json` defines a three-node `source -> transform -> sink` workflow with explicit capacity-2 edges.
- `examples/native-linear-etl.md` documents validate, explain, and run commands plus the expected metadata output.
- The example runs through the real CLI native registry path and writes 24 metadata records: six lifecycle records, four message boundary records, and 14 queue-pressure records.
- `README.md` now points to the example and reflects the current registry execution path.
- A CLI regression test includes the example workflow and asserts that it runs with three nodes, two edges, 24 records, and transform/cleaned metadata.

`pureflow-core` now emits structured queue-pressure metadata:

- `QueuePressureRecord` captures input/output direction, boundary kind, port ID, node context when available, connected edge count, total bounded capacity, and observable queued count.
- `MetadataRecord::QueuePressure(...)` serializes to stable JSON with `record_type: "queue_pressure"`.
- `PortsOut` records reserve attempts, reserve-ready, reserve-full, send-committed, and send-dropped observations without exposing raw runtime channel types.
- `PortsIn` records receive attempts, receive-ready, receive-empty, and receive-closed observations, including queued counts on input channels.
- `pureflow-runtime::run_node_with_observers(...)` now attaches node context to port handles before execution, so queue-pressure metadata includes workflow, node, execution, and cancellation context.
- The tracing feature labels queue-pressure metadata as `queue_pressure`.
- CLI explain output and the native linear ETL example documentation now mention lifecycle, message, and queue-pressure metadata. The native linear ETL example now writes 24 records: six lifecycle, four message, and 14 queue-pressure records.
