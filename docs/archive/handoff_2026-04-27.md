## What Changed

The most important implementation change is that workflow execution is now supervised concurrently rather than sequentially, while still keeping the public `NodeExecutor` API borrowed and Pureflow-owned:

- all workflow edges are wired before execution starts
- node futures are polled concurrently
- the first node failure requests workflow cancellation for siblings
- terminal results are still drained so errors are not lost

The core port boundary also gained async multi-input receive support:

- `PortsIn::recv_any` returns the first ready packet across declared inputs
- cancellation returns a typed `PortRecvError::Cancelled`
- fully closed inputs resolve to `Ok(None)`

The first WASM adapter crate now exists:

- `pureflow-wasm` owns the Wasmtime dependency and keeps it out of `pureflow-core`
- the MVP ABI is `pureflow:batch@0.1.0` / `pureflow-node` under `crates/pureflow-wasm/wit/`
- `WasmtimeBatchComponent` compiles a guest component, instantiates an import-free Component Model world, calls `batch.invoke`, and decodes dynamic Component Model values back into `BatchOutputs`
- typed `wasmtime::component::bindgen!` was rejected for the host adapter because its generated expansion contains unsafe internals and the workspace forbids unsafe code
- output port allowlist validation still belongs above the adapter, as recorded in the Claude WASM responses
- WASM/process contracts now reject declared effects that the current strict boundary cannot enforce; native contracts still accept effects as advisory metadata
- `pureflow-wasm` validates `NodeCapabilities` before component construction and rejects any effect capability because the MVP WIT world imports no host effects
- `pureflow-wasm` now has a checked `mixed_pipeline` example showing a native source, batch-backed WASM boundary node, and native sink wired through `pureflow-engine`
- the example uses an in-process `BatchExecutor` stand-in so the repo remains self-contained until a compiled guest component fixture lands
- `pureflow-core` now exposes `PacketPayload::Arrow(arrow_array::RecordBatch)` behind the `arrow` feature
- default builds remain Arrow-free; Arrow support is enabled explicitly with `--features arrow` or `--all-features`
- `RecordBatch` is `PartialEq` but not `Eq`, so Arrow-enabled payload containers do not derive `Eq`; default builds keep the existing `Eq` derives
- `pureflow-wasm` rejects Arrow or other non-WIT payloads at the WIT 0.1 boundary instead of coercing them into bytes/control
- `pureflow-workflow-format` now exists as the parser-free raw workflow boundary
- raw workflow documents require `pureflow_version = "1"` and reject missing or unsupported versions with typed errors
- raw workflow conversion routes identifiers, edge capacities, duplicate ports, unknown nodes/ports, and cycles through the existing domain constructors and validation errors
- canonical JSON support now lives in `pureflow-workflow-format` through `raw_workflow_from_json_str`, `workflow_from_json_str`, and `raw_workflow_to_json_string`
- JSON parsing preserves typed missing/unsupported `pureflow_version` diagnostics by decoding into raw structs before domain validation
- JSON raw structs reject unknown fields and round-trip through pretty JSON serialization
- TOML/YAML parser dependencies have not been added yet; those parser beads can layer on the same raw structs without polluting `pureflow-workflow`

## Question Responses

Claude responses in [docs/questions/responses/](/home/ben/code/pureflow/docs/questions/responses) have been reviewed and folded into the codebase where they had immediate action:

- Identifier length remains capped by raw UTF-8 bytes; `MAX_IDENTIFIER_LEN` now documents the transport/storage rationale.
- Contract schema compatibility remains exact equality while `SchemaRef` is opaque.
- Capability descriptors remain required for every workflow node; `NodeCapabilities::native_passive` covers the no-effects native case.
- Metadata message records stay delivery/drop specific; drained or disconnected port observations should land later as lifecycle/port-state metadata, not as message records.
- WASM batch executors accept empty inputs, stay topology-blind, and remain synchronous for the Wasmtime MVP; cancellation should be adapter-specific.
- WASM guest ABI uses Component Model/WIT first. Host pull/push was rejected, and core-Wasm export remains a documented fallback only if future tooling blocks WIT.
- `pureflow-wasm` uses dynamic Component Model `Val` invocation rather than host-side typed bindgen to preserve the workspace zero-unsafe policy.
- Strict boundaries currently support no host effects. Future WASI/process work should add explicit allowlist entries before granting `EffectCapability` values to non-native nodes.
- A real guest fixture still needs a build path or checked binary artifact; the current mixed example proves the host-owned batch integration without depending on `wasm32-wasip2` availability.
- Arrow support uses Apache `arrow-array` / `arrow-schema` directly. `tonbo-io/typed-arrow` remains a candidate for a future typed-node ergonomics layer, not the core runtime payload representation.
