# Documentation Architecture and Information Hierarchy

## Purpose

This document defines where Pureflow documentation should live, who it is for, and how it is produced.

## Output Surfaces

Pureflow documentation is split into three output surfaces:

1. Rust API docs generated by `cargo doc`.
2. Human architecture guide generated by Quarto as HTML and PDF.
3. LLM retrieval notes as compact Markdown reference files.

These surfaces are intentionally separate so each can optimize for a different reader and format.

## Canonical Structure

### API docs

- Source of truth: Rust source and rustdoc comments in `crates/`.
- Build command: `cargo doc --workspace --no-deps`.
- Output: local Rustdoc site under Cargo `target` output.

### Human guide

- Source of truth: Quarto project under `docs/architecture-guide/`.
- Inputs: narrative markdown plus optional diagrams/assets.
- Output: rendered HTML and PDF intended for human reading.

### LLM notes

- Source of truth: compact markdown files under `docs/llm/`.
- Constraints: short sections, stable facts, explicit headings, minimal prose.
- Output: repository markdown designed for retrieval/chunking.

## Directory Rules

- `docs/architecture-guide/`: long-form human narrative and publishing config.
- `docs/llm/`: retrieval-oriented topic notes.
- `docs/archive/`: historical handoffs and superseded snapshots.
- `docs/handoff_YYYY-MM-DD.md`: active day-level status snapshots.
- Existing operational docs in `docs/*.md` remain valid and can be incrementally moved when needed.

## Naming Conventions

- Date-stamped operational docs: `topic-YYYY-MM-DD.md`.
- LLM topic docs: `kebab-case-topic.md`.
- Avoid ambiguous names like `notes.md` or `misc.md`.

## Documentation Workflow

1. Update rustdoc comments in code for API behavior.
2. Update or add Quarto guide pages for human narrative and architecture context.
3. Update LLM notes when stable architectural facts change.
4. Regenerate outputs (`cargo doc`, Quarto render) as needed.

## Ownership and Update Trigger

- API docs change with code changes.
- Human guide changes with architecture or product communication changes.
- LLM notes change only when stable facts, contracts, or boundaries change.

## Near-Term Migration Plan

1. Keep current `docs/*.md` files in place.
2. Populate `docs/architecture-guide/` with foundational architecture guide pages.
3. Populate `docs/llm/` with core architecture fact sheets.
4. Move documents only when there is clear benefit and links are updated in the same change.
