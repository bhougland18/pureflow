## Recent Work

`pureflow-introspection` now owns pure workflow and contract introspection:

- crate added at [crates/pureflow-introspection/src/lib.rs](/home/ben/code/pureflow/crates/pureflow-introspection/src/lib.rs)
- workspace membership and dependency aliases added in [Cargo.toml](/home/ben/code/pureflow/Cargo.toml)
- `introspect_workflow` validates workflow contracts and capabilities before projection
- `WorkflowIntrospection` exposes workflow id, node views, and edge views
- node views include ports, execution mode, enforcement level, determinism, retry disposition, and effect capabilities
- port views include direction, schema, and port capability direction
- edge views include source, target, capacity, and source/target schemas
- `EnforcementLevel` reports native nodes as advisory and WASM/process nodes as strict

The implementation uses a separate crate rather than `pureflow-core::introspection`
because `pureflow-contract` already depends on `pureflow-core`; placing
contract-aware introspection in core would create a dependency cycle.

`pureflow-introspection` now also has serde-gated JSON rendering:

- feature added as `pureflow-introspection/serde`
- `workflow_introspection_to_json_value` returns a stable `serde_json::Value`
- `workflow_introspection_to_json_string` returns pretty-printed JSON for CLI and AI consumers
- JSON labels are explicit snake-case strings rather than derived enum names
- `IntrospectionJsonError` keeps JSON encoder failures typed
- inline `insta` snapshot coverage locks the current AI-facing JSON shape

Workspace dependency added:

- `insta = "1.47.2"` for introspection JSON snapshot tests

`pureflow-cli` now exposes the first real workflow commands:

- `pureflow validate <workflow.json>` parses canonical JSON workflow files and reports workflow id, node count, and edge count
- `pureflow inspect <workflow.json>` parses the same workflow and emits pretty introspection JSON
- `inspect` currently infers native-passive, no-schema contracts from workflow topology because there is not yet an external contract file format
- CLI errors preserve typed workflow JSON, contract, capability, and introspection JSON failures internally before formatting for stderr
- the old empty-workflow runtime scaffold was removed from the CLI
- unit tests cover valid summary output, inspect JSON output, and malformed JSON rejection

`pureflow-workflow-format` now supports TOML behind an explicit feature:

- workspace dependency added as `toml = "0.9.8"`
- crate feature added as `pureflow-workflow-format/toml`
- `raw_workflow_from_toml_str` decodes human-authored TOML into the existing raw workflow structs
- `workflow_from_toml_str` converts TOML directly into validated domain workflows
- `WorkflowTomlError` separates TOML decode failures from raw workflow format/domain failures
- TOML tests cover valid workflow decoding, missing version diagnostics, malformed TOML, and unknown-field rejection

`pureflow-workflow-format` now supports YAML behind an explicit feature:

- workspace dependency added as `serde_yml = "0.0.12"`
- crate feature added as `pureflow-workflow-format/yaml`
- `raw_workflow_from_yaml_str` decodes YAML into the existing raw workflow structs
- `workflow_from_yaml_str` converts YAML directly into validated domain workflows
- `WorkflowYamlError` separates YAML decode failures from raw workflow format/domain failures
- YAML tests cover valid workflow decoding, missing version diagnostics, malformed YAML, and unknown-field rejection
- YAML parser maintenance risk is isolated behind the optional feature

`pureflow-core` now has a reproducible JSONL metadata sink:

- `metadata_record_to_json_value` projects typed metadata records into stable JSON without wall-clock timestamps or process-local fields
- `JsonlMetadataSink<W>` records one JSON object per line to any `Write + Send`
- `JsonlMetadataSink::create(path)` provides the file-backed run-log constructor
- `flush` and `into_inner` preserve explicit writer control for tests and CLI callers
- JSON labels are explicit snake-case strings for lifecycle and message boundary kinds
- tests cover stable message JSON shape, reproducible repeated JSONL lines, and writer error mapping to `CDT-META-001`
