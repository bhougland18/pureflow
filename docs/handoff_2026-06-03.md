## Completed Work

### 1. Watcher-Cancellation Workload Example

A runnable native executor example demonstrating a watcher-style node whose
normal terminal path is cancellation.

**Files:**

- `crates/pureflow-engine/examples/watcher_cancellation.rs`
- `examples/workloads/watcher-cancellation.md`
- `examples/workloads/watcher-cancellation.workflow.json`

**Topology:**

```
change-source.changes  →  watcher.changes
change-source.changes  →  shutdown-controller.changes
shutdown-controller.control  →  watcher.control
```

Three nodes, three edges, all at `capacity = 1`.

**How it works:**

`ChangeSource` emits four file-change byte messages. `ShutdownController`
drains the full change stream, then emits one control message encoding the
change count. `Watcher` receives both streams via `recv_any`; when it sees the
control message it returns `PureflowError::cancelled` — cancellation is the
expected clean shutdown path, not a failure.

**Key design points:**
- Demonstrates `recv_any` for multi-port interleaved receive.
- Demonstrates `PureflowError::cancelled` as a normal terminal outcome.
- Asserts that `WorkflowTerminalState::Cancelled` with `cancelled_node_count == 1`
  is the expected summary — not an error to be fixed.
- Metadata output includes `node_cancelled` lifecycle records, error records
  for `CDT-CANCEL-001`, message records, and queue-pressure records.

**Run it:**

```sh
cargo run -p pureflow-engine --example watcher_cancellation
```

Expected terminal output ends with `terminal state: cancelled`.

### 2. Rules Engine — Architectural Design

No code was written. The deliverable is the proposal document and a
topic-specific handoff:

- `docs/archetecture/rules-engine-proposal.md`
- `docs/archetecture/rules-engine-handoff.md`

**What the rules engine is:**

A first-class native node type (`RuleNode`) and companion `pureflow-rules`
crate that moves conditional packet-routing logic from opaque executor code
into declared, named, serializable rule sets. Rules are data — validated at
load time, visible to `pureflow-cli inspect` and `pureflow-cli explain`,
and auditable in the metadata JSONL.

**Key decisions made in this session:**

1. **Runtime is asupersync, not tokio.** All async signatures take `cx: &Cx`.
   The evaluator returns `Outcome<RuleDecision, RuleError>` — not `Result`.
   `RuleNode` implements the asupersync multi-phase cancel protocol. Port
   sends use reserve/commit. Tests run under the asupersync deterministic
   lab runtime.

2. **`EvalOutcome` renamed to `RuleDecision`** to avoid collision with
   asupersync's `Outcome<T, E>` type.

3. **Conditions reach Pureflow context, not just payload.** The condition
   language was extended with tag conditions (`TagEq`, `TagExists`),
   provenance conditions (`SourceNode`, `ArrivedOnPort`, `HopCountGt/Lte`),
   and execution context conditions (`WorkflowIs`, `ExecutionMetadataEq`).
   `EvalContext` carries `PacketMetadata`, `WorkflowContext`, `NodeContext`,
   and asupersync `Budget` alongside the payload.

4. **`ConditionTrace` per condition in `RuleEvalRecord`.** Every condition
   evaluated during a rule check is recorded: which condition, what it saw,
   whether it matched, which surface it drew from (payload / tag / provenance /
   execution context).

5. **`RuleEvalRecord` emitted in asupersync finalize phase.** Guaranteed, not
   best-effort. A cancelled `RuleNode` mid-packet still emits the record.

6. **`RuleSetSource` trait for pluggable rule loading.** `pureflow-workflow-format`
   resolves `rule_set_ref` URIs through the trait rather than hardcoded
   filesystem reads. Ships with `LocalFsSource` and `EmbeddedSource`.

7. **Rule sync (GuardianDB / Iroh) is explicitly app-layer.** The user intends
   to use GuardianDB (https://github.com/wmaslonek/guardian-db) to deliver
   rule set updates to deployed apps without app-store resubmission. Pureflow
   defines the `RuleSetSource` boundary; the GuardianDB adapter is the
   application's concern. `pureflow-rules` has no dependency on GuardianDB
   or Iroh.

**13 beads identified in §11 of the proposal.** None filed yet.

**Open questions documented in `rules-engine-handoff.md`:**
- URI scheme registration mechanism for custom `RuleSetSource` implementations.
- `AllMatches` strategy collision behavior when two rules both match with
  terminal actions targeting different ports.
- `ConditionTrace` volume management under `AllMatches` for large rule sets.
- asupersync version pin to confirm before writing any `pureflow-rules` code.
